3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
Назва системи: Instagram
Розробник: Кевін Систром та Майк Крігер 
Призначення системи:  Instagram - це соціальна мережа та цифрова медіаплатформа для створення, публікації й поширення мультимедійного контенту та онлайн-комунікації між користувачами. Платформа використовується як для особистого спілкування, так і для маркетингу, просування брендів та ведення комерційної діяльності. Основні функції системи, які формують її складність, включають публікацію фото, відео та Reels, персоналізовану стрічку новин, Stories, Direct-повідомлення, а також системи пошуку та рекомендацій контенту.
3.2 Архітектурний стиль системи
Мікросервісна та подієво-орієнтована архітектури.
По-перше, мікросервісна архітектура забезпечує незалежне масштабування сервісів — якщо зростає навантаження на завантаження медіа, масштабується лише Media Service, а не вся система. По-друге, вона забезпечує ізоляцію відмов: збій одного сервісу не впливає на роботу інших. По-третє, дає технологічну гнучкість, оскільки різні сервіси можуть використовувати різні мови програмування та бази даних.
Водночас подієво-орієнтована архітектура забезпечує слабке зв’язування між сервісами, оскільки вони взаємодіють не напряму, а через події. Також вона дозволяє обробляти події асинхронно, що підвищує продуктивність системи та зменшує затримки для користувача.
3.3 Основні компоненти системи
Розглянемо архітектуру Instagram.
Перший рівень — інтерфейс користувача. Instagram підтримує три клієнтські платформи. Нативний iOS-застосунок написаний на Swift та Objective-C з використанням UIKit. Android-клієнт — на Kotlin та Java. Вебверсія побудована на React.js як односторінковий застосунок. 
Другий рівень — мікросервісна архітектура. Вхідною точкою до системи є API Gateway. Він приймає всі запити від клієнтів і виконує:
* автентифікацію та авторизацію;
* перевірку токенів доступу;
* rate limiting (обмеження кількості запитів);
* маршрутизацію до відповідних внутрішніх сервісів;
Для захисту та керування сесіями використовується окремий Authentication Service, який підтримує OAuth 2.0, двофакторну автентифікацію та виявлення підозрілої активності.
Кожен сервіс відповідає за окрему функціональну область і може масштабуватись незалежно. Розглянемо основні сервіси:
* Feed Service формує персоналізовану стрічку новин. Сервіс аналізує підписки користувача, лайки, коментарі, час перегляду контенту, взаємодію з профілями та соціальний граф користувача. На основі цих даних алгоритми машинного навчання визначають, які пости мають найвищий пріоритет для конкретного користувача.
* Media Service відповідає за обробку медіафайлів після завантаження. Сервіс виконує стиснення зображень, транскодування відео у кілька форматів та якостей, генерацію прев’ю, а також підготовку файлів для швидкої доставки через CDN. Для цього використовуються кодеки H.264 та AV1.
* User Service керує профілями користувачів і соціальним графом системи — підписками, підписниками та зв’язками між акаунтами. Саме цей сервіс визначає, хто може бачити контент користувача та які рекомендації профілів показувати.
* Messaging Service : Забезпечує доставку повідомлень у реальному часі через WebSocket-з'єднання, що підтримують постійний двосторонній канал між клієнтом і сервером,.
* Search & Explore: Використовує багатоетапну воронку ранжування для підбору контенту від акаунтів, на які користувач не підписаний, серед мільярдів можливих варіантів


Мікросервіси взаємодіють між собою через REST/gRPC API, а для асинхронних подій використовується подієва модель.
Третій рівень — зберігання даних. Instagram використовує підхід polyglot persistence — різні типи даних зберігаються у найбільш відповідних системах. PostgreSQL зберігає профілі та критичні дані. Cassandra обробляє величезні обсяги лайків, коментарів. Redis слугує кешем для швидкого доступу до стрічок та популярних даних. Власна графова база TAO від Meta зберігає мільярди зв'язків соціального графу. А всі медіафайли — оригінали та транскодовані варіанти — живуть в хмарних сховищах об’єктів.
Четвертий рівень — допоміжні сервіси. Apache Kafka є асинхронною шиною подій, яка розв'язує залежності між сервісами. CDN Meta Edge Network доставляє медіа з географічно найближчого вузла, що критично для швидкого завантаження контенту.
3.4 Взаємодія компонентів
Синхронна взаємодія реалізується через REST API та gRPC-запити, коли клієнт або один мікросервіс напряму звертається до іншого для отримання даних у реальному часі. Наприклад, запит на отримання профілю користувача або перевірку авторизації проходить через API Gateway і Authentication Service.
Взаємодії через WebSocket-з’єднання, яке використовується для постійного двостороннього зв’язку між клієнтом і сервером. Воно застосовується в основному в Instagram Direct для чату.
Асинхронна взаємодія реалізується через подієво-орієнтовану архітектуру з використанням платформи Apache Kafka. Вона дозволяє сервісам не залежати напряму один від одного.
На рисунку 3.1 можна побачити діаграму послідовності для завантаження медіа та публікацію їх.


  

Рисунок 3.1 - Діаграма послідовності для завантаження медіа


Коли користувач натискає «Поділитися», запит спочатку надходить до API Gateway, який робить перевірку сесії користувача та його прав на завантаження контенту та спрямвовує медіа дані та метадані до Media Service.
Сирий файл завантажується в об'єктне сховище, що забезпечує надійне зберігання бінарних об’єктів. Система запускає конвеєр обробки, який створює кілька версій відео або фото у різних роздільних здатностях (1080p, 720p, 480p) та форматах (AV1, H.264), щоб контент коректно відображався на будь-яких пристроях та при будь-якій швидкості інтернету. Інформація про пост (ID користувача, опис, посилання на файли в сховищі) записується в реляційну базу даних (PostgreSQL) для забезпечення цілісності.
Щоб користувачеві не доводилося чекати завершення всіх процесів, Media Service публікує подію «Пост створено» у чергу повідомлень ( Apache Kafka). Це дозволяє системі працювати асинхронно: користувач отримує підтвердження про успіх, а важкі фонові завдання тривають.
Feed Service отримує подію з Kafka та оновлює стрічки підписників, використовуючи гібридну стратегію:
* для звичайних користувачів (Fan-out on write): Пост негайно додається в кеш-стрічки всіх підписників у Redis. Це забезпечує завантаження стрічки менше ніж за 100 мс;
* для знаменитостей (Fan-out on read): Якщо у користувача мільйони підписників, система не оновлює мільйони кешів одночасно (щоб уникнути перевантаження), а обчислює стрічку підписника динамічно, коли той відкриває додаток.


Після завершення обробки медіафайли стають доступними через CDN. Файли кешуються на краях мережі (ближче до кінцевих користувачів), що дозволяє, наприклад, користувачеві в Токіо завантажувати медіа так само швидко, як і користувачеві в Нью-Йорку.
3.5 Масштабованість та надійність
Instagram забезпечує масштабованість завдяки горизонтальному масштабуванню та механізмам балансування навантаження. Вхідний трафік від користувачів спочатку потрапляє на балансувальники, які розподіляють запити між доступними серверами мікросервісів. Це дозволяє уникати перевантаження окремих вузлів і забезпечує стабільну продуктивність системи навіть при різкому зростанні кількості користувачів.
Важливим аспектом архітектури є відмовостійкість. Instagram побудований як розподілена система, де сервіси дублюються і працюють незалежно один від одного. У випадку виходу з ладу окремого сервісу система продовжує функціонувати без критичних збоїв. 
Для підвищення продуктивності активно застосовується кешування. Часто запитувані дані, такі як стрічка новин, профілі користувачів і популярний контент, зберігаються у швидкій пам’яті за допомогою Redis. Це значно зменшує навантаження на бази даних і скорочує час відповіді системи, що особливо важливо при великій кількості одночасних запитів.
Також у системі використовується реплікація даних. Інформація зберігається у кількох копіях на різних серверах і в різних датацентрах. Це забезпечує високу доступність даних, навіть якщо частина інфраструктури виходить з ладу. Крім того, реплікація дозволяє розподіляти навантаження на читання між кількома вузлами.
Instagram працює в умовах масштабної хмарної інфраструктури з використанням контейнеризації сервісів. Мікросервіси розгортаються у контейнерах і можуть автоматично масштабуватися залежно від навантаження. Медіа-контент зберігається у хмарному сховищ об’єктів, а доставка даних користувачам здійснюється через глобальну CDN-мережу.
3.6 Приклади програмної реалізації
Назва інструменту та версія: Google AI Studio(Gemini 3.1 Pro)
Мета використання: Отримання програмного коду для демонстрації Event-driven архітектури.
Внесок автора: Написання промпту для генерації коду та вибір архітектурного принципу.
Програмний код та запит до ШІ для генерації коду наведено в додатку В.
        Post Service  імітує обробку запиту від користувача на публікацію посту. Створюється стандартизований об'єкт події. Він містить метадані (унікальний eventId, тип події eventType: 'PostCreated', часову мітку) та безпосередньо самі дані (ID поста, ID автора, URL зображення). Сервіс підключається до брокера та відправляє сформовану подію у топік post-events. Після успішної відправки події в Kafka, сервіс завершує свою роботу 
        Notification Service працює безперервно і відповідає за доставку push-сповіщень підписникам після створення нового допису. Сервіс підключається до Kafka як підписник, вказуючи ідентифікатор групи (groupId: 'notification-group'). Це дозволяє масштабувати систему: якщо запустити кілька екземплярів цього сервісу, Kafka автоматично розподілить навантаження між ними (кожна подія буде оброблена лише одним екземпляром з групи). Сервіс підписується на топік post-events з параметром fromBeginning: true. Це забезпечує відмовостійкість: якщо сервіс був тимчасово вимкнений, після перезапуску він прочитає і обробить всі події, які пропустив. При отриманні події PostCreated, сервіс запускає функцію sendPushNotifications. 
        Реалізований підхід вирішує ряд класичних проблем монолітних систем:
* слабке зв'язування: Post Service та Notification Service повністю незалежні. Зміни в логіці розсилки сповіщень ніяк не вплинуть на процес створення постів;
* висока чуйність інтерфейсу: Користувачу не потрібно чекати закінчення розсилки сповіщень. Запис у Kafka займає мілісекунди, після чого користувач може продовжувати користуватися додатком;
* відмовостійкість: Якщо Notification Service "впаде", система не втратить дані. Post Service продовжить успішно створювати пости і складати події в Kafka. Коли Notification Service відновиться, він продовжить роботу з того місця, де зупинився;
* розширюваність: Щоб додати нову бізнес-функцію (наприклад, алгоритм машинного навчання для аналізу фото), не потрібно змінювати існуючий код. Достатньо створити новий сервіс і підписати його на той самий топік post-events.
